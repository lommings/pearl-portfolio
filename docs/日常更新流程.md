## 📝 日常工作流程

### 方案 A：撰寫新文章

```bash
# 1. 在專案目錄建立新文章
src/content/blog/my-new-story.md

# 2. 撰寫內容（用任何文字編輯器）
---
title: '我的新故事'
description: '這是一個有趣的故事'
pubDate: 'Feb 24 2026'
heroImage: '../../assets/blog-placeholder-1.jpg'
category: '奇幻小說'
tags: ['冒險', '魔法']
---

故事內容...

# 3. 本地預覽（可選，確認效果）
npm run dev
# 開啟 http://localhost:4321 查看

# 4. 滿意後，推送到 GitHub
git add .
git commit -m "新增文章：我的新故事"
git push

# 5. 等待 1-2 分鐘，Cloudflare 自動部署完成
# 6. 訪問 https://pearl-portfolio.pages.dev 查看更新
```

---

### 方案 B：修改現有文章

```bash
# 1. 編輯文章
# 例如：修改 src/content/blog/demo-story.md

# 2. 推送更新
git add .
git commit -m "修正文章：修改標題和內容"
git push

# 3. 等待自動部署完成
```

---

### 方案 C：批次更新多篇文章

```bash
# 1. 修改多個檔案
# - 新增 3 篇文章
# - 修改 2 篇舊文章
# - 更新圖片

# 2. 一次性推送
git add .
git commit -m "批次更新：新增 3 篇文章，修正 2 篇舊文章"
git push

# 3. 只會觸發一次部署（不是多次）
```

## 📝 提交訊息（Commit Message）建議

### 好的提交訊息

✅ `新增文章：龍之谷的傳說`  
✅ `修正文章：修改錯字和標點符號`  
✅ `更新：加入 3 張插圖到冒險故事`  
✅ `修復：修正側邊欄連結錯誤`  

### 不好的提交訊息

❌ `update`  
❌ `fix`  
❌ `123`  
❌ `修改`  

**原則：**
- 清楚說明做了什麼
- 使用中文（方便自己記錄）
- 簡短但具體

## 🔍 部署失敗怎麼辦？

### 常見原因

1. **Markdown 格式錯誤**
   - frontmatter 缺少必要欄位
   - YAML 格式錯誤

2. **圖片路徑錯誤**
   - 圖片檔案不存在
   - 路徑寫錯

3. **語法錯誤**
   - 組件使用錯誤

### 解決步驟

1. **查看 Cloudflare 建置日誌**
   - Dashboard → Deployments → 點擊失敗的部署
   - 查看錯誤訊息

2. **在本地測試**
   ```bash
   npm run build
   ```
   - 如果本地建置成功，線上應該也會成功
   - 如果本地失敗，修正錯誤後再推送

3. **修正後重新推送**
   ```bash
   # 修正問題
   git add .
   git commit -m "修正建置錯誤"
   git push
   ```

## 💡 最佳實踐

### 1. 定期推送

**建議：**
- 寫完一篇文章就推送
- 不要累積太多變更

**原因：**
- 容易追蹤問題
- 出錯時容易回滾

---

### 2. 本地預覽後再推送

**流程：**
```bash
# 1. 寫文章
# 2. 本地預覽
npm run dev

# 3. 確認無誤後推送
git add .
git commit -m "..."
git push
```

**好處：**
- 避免推送有錯誤的版本
- 確保排版正確

---

### 3. 使用有意義的提交訊息

**為什麼重要：**
- 方便日後查找
- 部署歷史一目了然
- 出問題時容易找到原因
